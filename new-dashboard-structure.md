# /qianchuan-dashboard 页面结构拆解

## 一、整体页面结构

`/qianchuan-dashboard` 是一个独立的新页面，用于复刻目标截图里的「巨量千川直播大屏 / 全域直播大盘」。它不应继续沿用旧 `/dashboard` 的罗盘复盘页面骨架。

页面整体建议采用「顶部导航 + 核心指标横卡 + 左侧主趋势图 + 右侧分析卡片栏」结构。

### 1. 顶部导航区

顶部导航区承担品牌识别、直播状态、数据口径和页面操作能力。

建议拆成三部分：

- 左侧：抖音电商 / 巨量千川标识。
  - 展示「抖音电商」品牌。
  - 展示「巨量千川」品牌。
  - 两者之间保持清晰分隔，体现目标页已经进入千川经营大盘语境。

- 中间：公告条。
  - 展示全域直播大屏升级、数据口径、使用说明等提示。
  - 支持公告分页，例如 `1 / 2`。
  - 支持关闭按钮。

- 右侧：直播与数据操作信息。
  - 店铺名。
  - 直播状态，例如「直播中」。
  - 直播时长，例如「2小时28分钟」。
  - 开播时间。
  - 数据口径说明。
  - 刷新数据按钮。
  - 全屏按钮。

### 2. 顶部核心指标区

顶部核心指标区是一张大横向蓝色指标卡，用于展示全域直播经营的核心结果。

指标卡应横向铺开，占据页面主宽度，并弱化旧 `/dashboard` 那种分散小卡片结构。

需要包含以下指标：

- GPM。
- 整体消耗。
- 净成交 ROI。
- 净成交金额。
- 观看成交转化率。
- 实时在线人数。
- 曝光观看率。
- 直播间整体观看人数。

视觉上建议：

- 重点突出「整体消耗」「净成交 ROI」「净成交金额」。
- 其余指标作为同一横卡内的次级指标排布。
- 使用深蓝渐变背景、白色大数字、浅色指标名称。
- 指标名称可以带虚线下划线，模拟目标截图中的可解释指标样式。

### 3. 左侧主趋势图区

左侧主趋势图区是页面主体，用于展示全域直播大盘的时间趋势。

需要包含：

- 顶部 Tab：
  - 整体趋势。
  - 素材表现。

- 指标开关：
  - 综合成本。
  - 净成交金额。
  - 综合 ROI。
  - 消耗。
  - 整体成交金额。
  - 整体支付 ROI。

- 右上角工具：
  - 5分钟粒度。
  - 筛选按钮。
  - 可选的放大 / 全屏图表按钮。

- 双轴趋势图：
  - 左轴用于金额类指标，例如净成交金额、整体成交金额。
  - 右轴用于消耗、ROI 或其他比例类指标。
  - 支持多条折线。
  - 支持面积填充。
  - 支持 hover tooltip。
  - 支持横向时间轴。

- 底部事件时间线：
  - 投放动作。
  - 调控动作。
  - 每类动作以独立行展示。
  - 使用不同颜色圆点标记动作发生时间。
  - 支持后续扩展「新增调控」入口。

### 4. 右侧成交渠道构成卡片

右侧第一张卡片用于展示成交来源拆解。

需要包含：

- 环形图。
  - 中心显示核心口径，例如「成交金额」。
  - 不同颜色表示不同渠道。

- 渠道占比列表。
  - 渠道名称。
  - 占比。
  - 颜色标识点。

目标截图中的渠道示例包括：

- 直播推荐。
- 短视频及图文引流。
- 个人主页及店铺&橱窗。
- 抖音商城推荐。
- 头条西瓜。

### 5. 右侧直播间核心漏斗卡片

右侧第二张卡片用于展示直播间核心转化链路。

需要包含：

- 分层漏斗条。
  - 每一层用横向条表示。
  - 条宽可按数值或相对比例缩放。

- 每层数据。
  - 直播间整体曝光次数。
  - 直播间观看次数。
  - 商品点击次数。
  - 成交订单数。

- 转化率箭头。
  - 在层级之间标注转化率。
  - 可在右侧使用连线或箭头展示每一步转化关系。

该模块的目标不是复盘直播画面，而是解释从曝光到成交的转化效率。

### 6. 右侧直播实时评论卡片

右侧第三张卡片用于展示直播实时评论。

需要包含：

- 评论列表。
- 用户名。
- 评论内容。
- 时间。
- 标签。

评论可按时间倒序展示。标签用于标识评论类型，例如：

- 直播画面。
- 商品咨询。
- 价格问题。
- 物流问题。
- 异常反馈。

该模块虽然展示评论，但它服务于实时经营观察，不应复用旧 `/dashboard` 的右侧直播预览和回放结构。

## 二、数据结构设计

建议新增 `qianchuanData` 对象，作为 `/qianchuan-dashboard` 的初始数据模型。

示例结构：

```js
export const qianchuanData = {
  basicInfo: {
    shopName: "青岛外贸8号店",
    liveStatus: "直播中",
    liveDuration: "2小时28分钟",
    startTime: "2026-06-20 06:45:31",
    dataStatus: "数据已校验",
  },

  topMetrics: {
    gpm: 1886.84,
    totalCost: 338.15,
    netRoi: 21.44,
    netGmv: 7250.0,
    viewToPayRate: 2.73,
    onlineUserCount: 30,
    exposureViewRate: 8.93,
    totalViewCount: 3703,
  },

  trendData: [
    {
      time: "06/20 06:45",
      netGmv: 0,
      cost: 1.2,
      roi: 0,
      totalGmv: 0,
      eventType: "投放动作",
    },
  ],

  channelComposition: [
    {
      channelName: "直播推荐",
      value: 4352,
      percent: 60.09,
    },
  ],

  funnelData: [
    {
      name: "直播间整体曝光次数",
      value: 38466,
      conversionRate: 9.22,
    },
  ],

  comments: [
    {
      user: "凡***",
      content: "海口两套运费不减一个吗",
      time: "09:12",
      tag: "物流问题",
    },
  ],
};
```

字段说明：

- `basicInfo`：页面顶部基础信息。
  - `shopName`：店铺名。
  - `liveStatus`：直播状态。
  - `liveDuration`：直播持续时间。
  - `startTime`：开播时间。
  - `dataStatus`：数据状态或口径说明。

- `topMetrics`：顶部核心指标。
  - `gpm`：GPM。
  - `totalCost`：整体消耗。
  - `netRoi`：净成交 ROI。
  - `netGmv`：净成交金额。
  - `viewToPayRate`：观看成交转化率。
  - `onlineUserCount`：实时在线人数。
  - `exposureViewRate`：曝光观看率。
  - `totalViewCount`：直播间整体观看人数。

- `trendData`：主趋势图数据。
  - `time`：时间点。
  - `netGmv`：净成交金额。
  - `cost`：消耗。
  - `roi`：ROI。
  - `totalGmv`：整体成交金额。
  - `eventType`：事件类型，例如投放动作、调控动作。

- `channelComposition`：成交渠道构成。
  - `channelName`：渠道名称。
  - `value`：渠道成交值或成交贡献。
  - `percent`：渠道占比。

- `funnelData`：直播间核心漏斗。
  - `name`：漏斗层级名称。
  - `value`：该层级数值。
  - `conversionRate`：到下一层或相邻层的转化率。

- `comments`：直播实时评论。
  - `user`：用户名。
  - `content`：评论内容。
  - `time`：评论时间。
  - `tag`：评论标签。

## 三、文件结构设计

建议新建以下文件，并保持职责边界清晰。

### `qianchuanData.js`

负责千川大盘的数据模型和默认数据。

建议包含：

- `qianchuanData` 默认对象。
- 顶部指标数据。
- 趋势图数据。
- 渠道构成数据。
- 漏斗数据。
- 评论数据。
- 后续可扩展的数据归一化函数。

### `qianchuanDashboard.js`

负责 `/qianchuan-dashboard` 页面渲染和交互组织。

建议包含：

- 页面入口函数，例如 `renderQianchuanDashboard(root)`。
- 顶部导航区渲染。
- 顶部指标区渲染。
- 主趋势图区容器渲染。
- 右侧三张卡片渲染。
- 页面级事件绑定。
- 调用 `qianchuanChart.js` 完成图表绘制。

### `qianchuanChart.js`

负责目标页所有图表绘制。

建议包含：

- 双轴趋势图绘制。
- 趋势图 hover 和 tooltip。
- 底部事件时间线绘制。
- 成交渠道构成环形图绘制。
- 直播间核心漏斗图绘制。
- 图表 resize 处理。

如果后续引入图表库，也应优先让该文件承担适配层，而不是把图表逻辑散落到页面渲染文件里。

### `qianchuan.css`

负责 `/qianchuan-dashboard` 的独立样式。

建议包含：

- 页面背景。
- 顶部导航。
- 公告条。
- 核心指标横卡。
- 主趋势图区。
- 右侧分析卡片。
- 评论列表。
- 图表 tooltip。
- 响应式适配。

该文件应尽量避免依赖旧 `/dashboard` 的具体 class 结构，防止新页面被旧页面样式牵制。

## 四、旧代码复用策略

### 可以复用的旧代码

可以复用基础能力和局部实现思路：

- `serve-local.mjs`：继续作为本地静态服务。
- `app.js` 的路由判断方式：新增 `/qianchuan-dashboard` 分支即可。
- `storage.js` 的 localStorage 读写、导入、导出能力。
- `formulas.js` 的数字、金额、百分比格式化能力。
- 旧 canvas 趋势图中的基础绘图思路，例如坐标换算、折线绘制、hover 定位、tooltip 更新。
- 部分深色背景、蓝色高亮、按钮、标签、卡片阴影等视觉变量或设计思路。

### 不能复用的旧代码

不能复用旧 `/dashboard` 的主结构和复盘场景模块：

- 旧顶部指标卡布局。
- 旧右侧直播预览模块。
- 旧「综合趋势」图表结构。
- 旧 admin 表单结构。
- 旧「抖音电商罗盘直播大屏」整体页面骨架。
- 旧复盘 tab：综合趋势、流量分析、主播分析、流量诊断、引流短视频、违规情况。
- 旧场记回放语义：讲解、福袋、场记、预警等作为复盘主线的结构。

### 为什么不能直接在旧 `/dashboard` 上改？

不能直接改旧 `/dashboard`，核心原因是两者的信息架构不同。

旧 `/dashboard` 是「罗盘直播复盘页」，页面重点是单场直播回看、直播预览、评论回放、复盘趋势和场记事件。目标页是「巨量千川直播大屏 / 全域直播大盘」，页面重点是全域经营、投放消耗、ROI、成交渠道、核心漏斗和实时运营状态。

如果直接在旧 `/dashboard` 上改，会遇到几个问题：

- 旧右侧直播预览会占据目标页右侧分析卡片的位置。
- 旧指标模型与目标页的千川经营指标不一致。
- 旧趋势图围绕单场复盘，目标趋势图围绕全域经营和投放调控。
- 旧 admin 表单服务旧数据模型，无法自然承载千川数据结构。
- 旧页面样式和布局约束会不断影响新页面，导致目标页变成旧页面的局部换皮。

因此，新页面应该独立建立页面骨架，只复用底层工具能力。

## 五、实现顺序

建议按以下优先级推进。

1. 先做静态布局。
   - 新增 `/qianchuan-dashboard` 路由。
   - 建立顶部导航、核心指标区、左侧主趋势区、右侧三张卡片的静态 DOM。
   - 先保证布局比例、区域位置和目标截图一致。

2. 再做顶部指标。
   - 接入 `qianchuanData.topMetrics`。
   - 完成金额、百分比、人数等格式化。
   - 对齐目标截图中的主次层级和横向指标卡视觉。

3. 再做趋势图。
   - 接入 `qianchuanData.trendData`。
   - 绘制双轴趋势图。
   - 实现指标开关、5分钟粒度展示、hover tooltip。
   - 绘制底部投放动作和调控动作时间线。

4. 再做右侧三张卡片。
   - 成交渠道构成：环形图和渠道列表。
   - 直播间核心漏斗：漏斗条、每层数据、转化率箭头。
   - 直播实时评论：评论列表、用户名、内容、时间、标签。

5. 再做数据配置页。
   - 新增或扩展配置入口以支持千川数据模型。
   - 不建议直接复用旧 admin 表单结构。
   - 可以复用旧 admin 的保存、导入、导出交互思路。

6. 最后再考虑 React 迁移。
   - 当前阶段优先复刻目标截图和验证数据结构。
   - 如果页面复杂度继续上升，再考虑把 `/qianchuan-dashboard` 拆成 React 组件。
   - React 迁移不应阻塞当前静态复刻阶段。

## 总结

`/qianchuan-dashboard` 应作为独立的新页面建设。它的页面目标、结构和数据模型都应围绕「巨量千川直播大屏 / 全域直播大盘」重新设计。

旧 `/dashboard` 只适合作为底层能力参考，不适合作为新页面的主骨架继续改。
