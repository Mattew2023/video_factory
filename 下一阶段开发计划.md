# 下一阶段开发计划：自动录屏扫描与智能压缩

本计划用于本项目后续新对话接续开发。当前项目已有手动视频压缩工具和抖音合并压缩工具，下一阶段目标是把它演进成一个“录屏文件管家”：每天自动扫描指定目录，识别未压缩录屏，按文件大小、时长、码率等信息选择压缩策略，压缩成功并校验后再安全处理原文件。

## 一、核心目标

1. 每天自动扫描指定磁盘或目录中的录屏文件。
2. 自动识别“应该压缩但尚未压缩”的文件。
3. 根据文件大小、时长、分辨率、码率、编码格式判断压缩参数。
4. 压缩后生成带有“已压缩”标记的新文件。
5. 压缩成功并通过校验后，按安全策略处理原文件。
6. 保留日志和任务记录，方便追踪、重试和排查。

## 二、安全约束

必须遵守项目删除约束：

- 禁止批量删除文件或目录。
- 禁止使用 `del /s`、`del /f /s /q`、`rd /s`、`rmdir /s`、`Remove-Item -Recurse`、`Remove-Item -Force -Recurse`、`rm -rf`、`rm -r`。
- 需要删除文件时，只能一次删除一个明确路径的文件。
- 删除时禁止使用通配符，比如 `*` 或 `?`。
- 如果需要批量删除或递归删除目录，必须停止操作，请用户手动删除。

建议第一版不要直接永久删除原文件，而是：

1. 压缩成功后先校验新文件。
2. 校验通过后将原文件移入“待删除”目录，或移入回收站。
3. 后续再提供“逐个确认删除”的能力。

如果用户明确要求自动删除，也必须逐个明确路径处理，并且只在校验通过后执行。

## 三、MVP 范围

第一版先做“可靠自动压缩”，暂不做复杂 AI 内容理解。

功能范围：

- 配置扫描目录。
- 配置每天自动扫描时间。
- 扫描视频文件：`.mp4`、`.ts`、`.mkv`、`.mov`、`.avi`。
- 跳过文件名包含“已压缩”的文件。
- 跳过正在写入或最近刚修改的文件。
- 使用 `ffprobe` 读取视频元信息。
- 自动计算压缩目标。
- 使用 `ffmpeg` 压缩。
- 输出文件名增加“已压缩”。
- 写入本地任务记录。
- GUI 中显示待处理、处理中、成功、失败、跳过。
- 失败任务可重试。

暂不做：

- 自动分析录屏内容。
- 自动改名归档。
- 自动总结视频内容。
- 批量永久删除原文件。

## 四、推荐模块设计

建议从现有 Python/Tkinter 项目中逐步拆出以下模块：

```text
video_scanner.py
  扫描目录，识别候选视频文件。

media_probe.py
  调用 ffprobe，获取时长、码率、分辨率、编码格式、音频信息。

compression_policy.py
  根据元信息决定是否压缩，以及目标大小、视频码率、音频码率、编码器。

compression_runner.py
  调用 ffmpeg 执行压缩，解析进度，返回结果。

compression_records.py
  保存任务记录，避免重复处理。

safe_file_ops.py
  只做安全的单文件移动、重命名、删除或回收站操作。

auto_scan_app.py
  GUI 和定时任务入口。
```

如果第一步想少改动，也可以先在现有 `视频压缩工具.py` 基础上加“扫描目录”页，再逐步拆模块。

## 五、任务记录设计

建议使用本地 JSONL 或 SQLite。MVP 可以先用 JSONL，后续再迁移 SQLite。

每条记录包含：

```json
{
  "source_path": "D:/recordings/demo.mp4",
  "output_path": "D:/recordings/demo_已压缩.mp4",
  "source_size_bytes": 1234567890,
  "output_size_bytes": 456789000,
  "duration_seconds": 3600.2,
  "width": 1920,
  "height": 1080,
  "video_bitrate_kbps": 6000,
  "audio_bitrate_kbps": 160,
  "video_codec": "h264",
  "status": "success",
  "decision": "compress",
  "error": "",
  "created_at": "2026-07-02T10:00:00+08:00",
  "updated_at": "2026-07-02T10:20:00+08:00"
}
```

状态建议：

- `pending`
- `probing`
- `skipped`
- `compressing`
- `success`
- `failed`
- `verified`
- `source_moved_to_pending_delete`
- `source_deleted`

## 六、识别规则

候选文件条件：

1. 扩展名属于支持的视频格式。
2. 文件名不包含“已压缩”。
3. 文件大小大于配置阈值，默认 `500 MB`。
4. 最近修改时间超过稳定窗口，默认 `10 分钟`，避免处理录制中的文件。
5. 文件路径不在输出目录、待删除目录、临时目录中。
6. 历史记录中没有成功处理过同一路径和同一文件大小的文件。

跳过条件：

- 文件名包含“已压缩”。
- 文件过小。
- 文件正在写入。
- `ffprobe` 无法读取。
- 已经是低码率视频。
- 目标压缩后可能画质过低。

## 七、压缩策略

用 `ffprobe` 获取：

- 文件大小。
- 时长。
- 分辨率。
- 帧率。
- 视频码率。
- 音频码率。
- 视频编码格式。
- 音频编码格式。

基础判断：

```text
当前总码率 kbps = 文件大小字节 * 8 / 时长秒 / 1000
目标总码率 kbps = 目标大小字节 * 8 / 时长秒 / 1000
目标视频码率 kbps = 目标总码率 - 目标音频码率
```

建议策略：

- 1080p 录屏：目标视频码率约 `1500-3000 kbps`。
- 720p 录屏：目标视频码率约 `900-1800 kbps`。
- 画面变化少的办公录屏：可以更低。
- 原始视频已经低于 `800 kbps`：默认跳过。
- 算出的目标视频码率低于 `300 kbps`：提示风险或跳过。
- 原文件大于 `1 GB`：优先压缩到 `950 MB` 以下。
- 原文件大于 `3 GB`：可考虑 H.265，但要保留兼容选项。

第一版编码建议：

- 优先使用 H.264：兼容性最好。
- 参数：`libx264 -preset medium -b:v <目标码率> -b:a <音频码率>`。
- 后续可增加 H.265：`libx265`，用于长期归档。

## 八、压缩成功校验

压缩完成后必须校验：

1. 输出文件存在。
2. 输出文件大小大于 `0`。
3. 输出文件可被 `ffprobe` 读取。
4. 输出时长与原始时长差距在可接受范围内，比如 `3 秒` 或 `1%`。
5. 输出文件大小小于原文件。
6. 输出文件名包含“已压缩”。

只有全部校验通过，才允许处理原文件。

## 九、原文件处理策略

推荐默认策略：

```text
压缩成功 -> 校验通过 -> 原文件移动到 待删除/ 目录 -> 记录状态
```

可选策略：

- 保留原文件。
- 移动到待删除目录。
- 移入系统回收站。
- 逐个明确路径永久删除。

不建议 MVP 默认永久删除。若实现永久删除，必须在 GUI 中清楚显示路径，并逐个文件执行。

## 十一、自动运行方案

MVP 可以先做软件内定时：

- 程序启动后根据配置定时扫描。
- 适合用户每天开着工具。

下一步再做 Windows 计划任务：

- 提供“创建计划任务”按钮。
- 每天指定时间启动工具并执行扫描。
- 计划任务只调用程序，不在脚本里做复杂删除。

## 十三、验收标准

MVP 完成时应满足：

- 能选择一个目录并扫描出候选视频。
- 能正确跳过名称包含“已压缩”的文件。
- 能跳过正在写入或最近修改的视频。
- 能读取视频时长、大小、分辨率、码率。
- 能自动给出压缩目标。
- 能批量排队压缩，但删除或移动原文件必须逐个明确路径执行。
- 能生成“已压缩”输出文件。
- 能校验输出文件。
- 能记录每个任务的状态。
- 工具关闭后再次打开，不重复处理已成功任务。
- 出错时不会删除原文件。
